# Unity UI 全盘职业美术

## 目标

用户只需要选择一次主题。之后接管项目的完整表现层：理解现有功能，做出专业美术判断，把所有界面、可见实体、状态和反馈统一成一套正式、清晰、可维护的视觉系统。

主题比较宽泛时，根据玩法、世界观、现有资产和目标平台自行收敛，不再让用户选择子方案。只有涉及品牌、授权、玩法、信息架构或不可逆操作时才暂停确认。

## 工作方式

开始前完整阅读 [全盘生产协议](references/design-production-protocol.md)，然后持续推进：

1. **盘清全项目**：用 Unity MCP 找到全部 UI、动态对象、资源来源、运行入口和功能契约，建立覆盖账本。
2. **建立主题系统**：定义背景、承载面、文字、操作、状态、核心实体和动效的统一规则，并映射到项目现有技术栈。
3. **先打好基础**：在核心玩法、内容浏览、弹窗系统三类界面上验证主题，确保不同信息密度下都成立。
4. **全量铺开**：逐个界面族修改真实 Prefab、Scene、UXML/USS、生成器、材质和动态来源。
5. **主动找错**：专门检查旧主题残留、低对比文字、像禁用态的按钮、异常背景、旧卡背、动态 Sprite 漏项和跨界面不统一。
6. **循环修复**：发现问题就修根因，再复查受影响范围和全项目；账本没有清零前不要停。
7. **完整验证**：检查功能、引用、输入、布局、状态、Console、测试、构建和性能，最后交给用户在本地 Unity 中做一次整体视觉验收。

一轮做不完就继续下一轮。轮次是代理在同一任务里的内部循环，不是让用户反复审批，也不要询问“是否继续”。

## 底线

- 默认覆盖所有用户看得见的内容，包括卡牌、卡背、牌堆、棋盘、背景、字体、图标、粒子、世界空间 UI、动态 Sprite、脚本生成对象以及加载、空、错误和极端状态。
- 保留玩法、数据绑定、输入、序列化引用、UnityEvent、脚本 API、动画路径、资源 key、存档和平台约束。需要改变 UX 时单独取得授权。
- 按视觉语义设计，不按 `Image`、`Button`、对象名或路径关键词机械换色。动态赋值不是保留旧主题的理由。
- 真实 Unity 源资产是唯一权威。HTML 只可按用户明确要求导出说明，不能作为设计或验收结果。
- Unity Editor 操作只使用 Unity MCP 或项目内受控 Editor API；禁止 `computer-use`、坐标点击和自建 MCP 传输层。
- 固定使用无图片模式：不生图、不看图、不截图、不录屏，不让工具返回图片，也不在对话中嵌入图片。
- 保护用户已有改动，不用破坏性 Git 操作，不随意 Apply/Revert/Unpack Prefab，不删除仍有引用的旧资产。

## 进度与续跑

把主题规则、覆盖账本、消费者映射、例外和验证结果保存在 `.unity-ui-frontend-engineer/`。每轮只汇报简短进展，然后自动继续。上下文压缩或任务恢复时从账本续跑，不重新询问主题，也不重复已完成工作。

## 什么时候才算完成

必须同时满足：

- 全部可见界面、实体、动态消费者和关键状态都已覆盖；
- 没有未登记的硬编码样式、旧主题污染或明显离群项；
- 文字、按钮和状态清晰可辨，功能、输入、资源、测试、构建和性能均已验证；
- 覆盖账本只剩已验证项或用户批准的例外；
- 用户完成最终本地视觉验收。

如果用户指出任何不统一、不可读、遗漏或不专业的问题，重新打开账本并继续修复循环，不要把问题留作“后续建议”。
