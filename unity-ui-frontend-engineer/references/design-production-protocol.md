# Unity UI 全盘生产协议

这份协议是执行清单。目标很简单：用户选一个主题，代理用 Unity MCP 把整个项目的表现层做完，并持续修到统一、清晰、可用。

## 1. 开始前

每次开始、Unity 重启或任务恢复时：

1. 读取 `mcpforunity://custom-tools`、`mcpforunity://instances`、`mcpforunity://project/info`、`mcpforunity://editor/state` 和所需 API；多实例时固定目标。
2. 读取 `.unity-ui-frontend-engineer/` 里的主题规则、覆盖账本、当前轮次和失败项。
3. 检查工作树、Console、测试、构建、性能、引用和 Prefab 结构基线，保留用户已有改动。
4. Unity 操作只走 Unity MCP 或项目内受控 Editor API。MCP 断开时保存进度并报告，不降级到 GUI 自动化。

始终使用无图片模式：不调用生图、看图、截图或录屏工具，不让 MCP 返回图片，不在回复中嵌入图片。现有位图只通过路径、GUID、尺寸、导入设置和引用关系处理。

## 2. 盘清范围

默认纳入所有用户能看到的表现：

- Scene、Canvas、Panel、Prefab、UXML/USS、模板、主题配置和生成器；
- HUD、菜单、弹窗、列表、Tab、Tooltip、Toast、遮罩、加载、空、错误和结果界面；
- 卡牌、卡背、牌堆、棋盘、物品、头像、标签、图标、背景、字体、光标和世界空间 UI；
- 材质、Shader、粒子、VFX、转场、动效和状态反馈；
- 工厂、池化、脚本生成对象、动态 Sprite、数据驱动图标和 Addressables/Resources；
- 正常、空、满、长文本、极限数字、加载、失败、禁用、悬停、按下、选中、危险、成功和输入焦点。

用 Play Mode 和真实入口确认运行时消费者。无法触发的状态保持未完成，并记录复现条件，不能靠静态层级猜测。

覆盖账本至少记录：权威源、运行路径、消费者、界面/实体族、视觉角色、动态来源、状态与输入、主题映射、正式资产、对比度、旧主题残留、证据、失败根因、修复轮次和状态。

## 3. 锁定主题

用户已经给出主题时直接接受，不再制作 A/B/C 子方案。根据玩法、世界观、现有资产、受众和平台，把主题写成一套项目专属规则：

- 主题命题和三个反目标；
- 背景、承载面、HUD、正文、强调、主/次操作、禁用、选中、危险、成功、遮罩、核心实体和反馈的角色；
- 字体、尺度、间距、轮廓、材质、图标、动效、状态、对比度和性能规则；
- 每个角色到真实 Unity 源和消费者的映射。

用户明确调用技能并选择主题，就视为授权完成整个表现层。只有品牌、授权、玩法、信息架构或不可逆操作需要额外确认。

## 4. 实施

先在三类代表界面上打好基础：

1. 核心玩法、HUD 和动态实体；
2. 大厅、导航、列表和内容浏览；
3. 弹窗、设置、确认、加载和错误状态。

这只是内部校准，不是新的用户审批门。基础成立后立即全量迁移。

每个界面族按同一节奏推进：

1. 写清背景、承载面、文字、操作和实体分别使用什么主题角色；
2. 修改真实 Prefab、Scene、UXML/USS、模板、生成器或动态赋值源；
3. 反查所有消费者和共享资产影响；
4. 运行对应状态、输入和 Play Mode 验证；
5. 更新覆盖账本并继续下一族。

批量工具只能执行已经确认的角色映射。禁止给所有 `Image`、`Button` 或 Panel 套同一个颜色和表面；也不能以“保护动态 Sprite”为由留下旧卡背、牌堆或数据驱动图标。

## 5. 审查和修复循环

全量迁移后必须单独进行一次“主动找错”，重点检查：

- 旧颜色、字体、Sprite、卡背、底图、边框、纹理、材质和装饰；
- 绕过主题系统的硬编码值、脚本和生成器；
- 浅底浅字、复杂背景裸文字、按钮像禁用、状态差异不清；
- 异常背景、异常深按钮、孤立字体、旧核心实体和跨界面离群值；
- 动态 Sprite、池化对象、数据驱动图标和极端状态漏项；
- 长文本、语言扩张、滚动、裁切、遮挡、安全区和点击命中问题。

发现问题时先修根因：主题规则错误就修规则，共享组件错误就修共享源，界面特例就做明确变体，动态问题就修生成器、工厂、池化或赋值源。修完后复查所有消费者和全局离群值。

出现新问题就继续下一轮，不设固定轮数，不询问用户是否继续，也不把未解决项写成后续建议。每轮更新完成项、失败项、根因、修改源、验证结果和下一轮队列。

## 6. 质量检查

### 可读性和状态

- 可计算背景上的正文对比度至少 4.5:1，大号文字和关键非文字控件至少 3:1。
- 复杂背景上的文字必须有稳定承载面。
- 检查字号、行距、字重、对齐、截断、换行、极端数字、语言扩张和安全区。
- 检查 Selectable 的 normal、highlighted、pressed、selected、disabled 和输入焦点。
- 刷新、购买、确认等关键操作不能看起来像禁用；危险、成功、选中和焦点不能只靠微小色差。

### 一致性

- 每个可见元素都要有视觉角色、主题映射、正式资产和消费者。
- 扫描 Prefab、Scene、UXML/USS、模板、生成器和脚本中的颜色、字体、Sprite、材质、状态值与旧主题资源。
- 非主题值必须修复或登记有充分理由的例外。
- 反查卡牌、卡背、牌堆、棋盘、物品、数据图标、工厂和池化对象，验证生成、复用、回收和异步加载路径。

### 工程

- 验证 Missing Script/reference、serialized field、UnityEvent、查询、动画、Variant、资源 key、存档和动态实例。
- 运行编译、EditMode/PlayMode 测试、Player/Addressables 构建、Console 和静态检查。
- 检查 uGUI 的 rebuild、批次、Atlas、Mask、overdraw；UI Toolkit 的 style/layout/repaint 和虚拟化；世界空间 UI 的相机、排序、Raycaster、像素密度和遮挡。
- 与写入前基线对比，区分既有问题和本次回归。

## 7. 完成

只有以下条件全部满足，才进入最终视觉验收：

- 覆盖账本没有待发现、待实现、审查失败或修复中的项目；
- 全部可见实体、动态消费者和关键状态已覆盖；
- 没有未登记硬编码样式、旧主题引用或明显离群值；
- 对比度、交互状态、布局、输入、资源、Console、测试、构建和性能通过；
- 无法运行的项目有真实阻塞原因和复验步骤，并且不影响完成结论。

整理一条覆盖全部真实界面族和关键状态的 Unity Game View 审阅路线，让用户做一次整体视觉验收。

用户指出任何不统一、不可读、遗漏或不专业的问题时，重新打开相关账本项，进入下一修复轮，修完后再做全局复查。只有用户明确视觉通过后才宣称完成。
