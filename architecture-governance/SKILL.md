# 架构治理

## 目标

面向 AI 生成、快速堆叠或一次性写成的代码，进行持续的架构降熵治理。核心工作是大面积识别并消除偶然复杂度、临时结构、重复知识、失控状态、边界泄漏和历史遗留，让代码在功能不变的前提下变得更少、更硬、更清晰，并拥有可长期维护的模块边界。

本技能不是代码格式化、普通清理或单次重构。它是无人看守时也必须持续推进的治理循环。

## 无限治理契约

1. 本技能启动后必须进入无限治理模式，不得把任务解释为“完成若干轮即可结束”的普通任务。
2. 完成若干轮、测试通过、提交成功、工作树干净、计划项暂时变少、暂时没有高价值目标，都不是停止条件。
3. 禁止因为“已完成 N 轮”“当前没有明显问题”“收益可能变小”“需要等待用户查看”发送 final；轮数只能作为进度记录，不能作为完成判断。
4. 每轮提交后必须立即开启下一轮侦察；如果本轮无高价值目标，记录检查范围和结论后，必须更换粒度、入口、模块或侦察切面继续下一轮。
5. 只要还能读取仓库、运行工具、缩小范围或更换侦察入口，就不得主动收尾。
6. 只能在以下场景发送 final：
   - 用户明确要求停止、暂停、只回答问题或不要继续执行。
   - 同一真实阻塞条件已经连续出现，且无法通过本地上下文、工具、更小切口或更换侦察入口继续推进。
   - 系统、权限、上下文或工具边界强制中断执行；此时必须说明治理未完成，并给出下一轮应从哪里继续。

## 总流程

1. 检查目标 git 工作树；无法证明用户改动不冲突时阻塞。
2. 创建治理分支，默认 `architecture-governance-<YYYYMMDD>`，除非用户指定。
3. 扫描项目规则、构建入口、测试入口、运行入口、核心领域代码和近期 AI 生成痕迹。
4. 创建或全量重写 `.architecture-governance/plan.md`。
5. 进入无限轮次循环。

## 轮次循环

1. 每轮固定开启 5 个侦察子代理，对应 5 个侦察方向，等待他们全部完成。
2. 结合所有侦察子代理输出和项目事实，整合并全量重写 `plan.md`。
3. 关闭所有侦察子代理。
4. 完成 `plan.md` 中所有本轮执行项，验证，复查 diff。
5. 提交本轮修改。
6. 立即进入下一轮侦察；禁止停在提交成功、工作树干净或无明显问题。

## 侦察子代理

1. 禁止使用全历史 fork；子代理不得继承主线程对话历史。
2. 子代理只能通过启动提示词接收任务；方向、范围、禁止事项和输出格式必须写入提示词。
3. 只做只读侦察，禁止修改文件。
4. 侦察依据为项目事实，必须给出文件、符号、调用路径、测试、运行行为或 diff 证据。
5. 每个子代理只侦察一个固定方向；同一轮必须覆盖全部 5 个方向，禁止重复方向或跳过方向。
6. 侦察禁止停留在表面命名、格式或风格，优先找出能降低代码熵、错误面或认知负担的高价值目标。
7. 子代理必须识别 AI 一次性代码常见熵源：重复实现、无主抽象、过宽 DTO、临时兼容、双路径逻辑、过度防御、状态散落、注释解释旧设计、测试锁死实现细节。

## 侦察输出契约

1. 启动提示词必须包含：目标仓库、侦察方向、只读约束、证据要求、输出格式。
2. 输出固定为：结论、证据、风险、建议改动、不建议改动。
3. 无发现时也必须说明已检查范围和“无高价值目标”。
4. 任一方向失败或缺失时，本轮不得进入实现，必须补齐该方向侦察。

## 提议管线

1. 所有发现先归类：bug、架构债、统一性漂移、死代码、文档漂移、AI 堆叠熵源。
2. 本轮执行项必须写明：现状、降熵目标、功能不变证据、改动策略、风险、验证、回滚边界。
3. bug 项必须有触发条件和回归测试；无法复现只能标记阻塞或不采纳。
4. 涉及用户可见行为、数据、异步/事务、公共 API 或资源序列化的项，缺少契约或测试时不得执行。
5. 实现后必须复审 diff：是否超出计划、是否迎合测试、是否引入新抽象、新状态、新历史遗留或新的双路径。

## 计划执行契约

1. `plan.md` 不是候选池，而是本轮执行清单。
2. `plan.md` 每轮必须全量重写，只保留当前轮次；已完成轮次以 git 提交历史为准，禁止持续追加。
3. `plan.md` 必须按 5 个侦察方向列出本轮全部发现、归类和状态。
4. 每个侦察发现必须标记为：本轮执行、不采纳、阻塞；禁止无说明丢弃。
5. 禁止用“下一轮候选”停放已确认发现；跨轮保留只能标记为阻塞。
6. 标记为本轮执行的项目必须全部完成，禁止只选择一个方向或一个子代理结论。
7. 不采纳必须基于核心原则、证据不足或功能不变风险；禁止因为工作量、提交粒度或偏好不采纳。
8. 阻塞必须写明缺少的事实、工具或决策。
9. 验证暴露的阻塞问题可以纳入本轮；非阻塞新发现不得顺手修改，必须进入下一轮 5 方向侦察。
10. 下一轮必须重新执行 5 个侦察方向，禁止直接沿用上一轮候选替代侦察。
11. 提交前必须复查 `plan.md` 和 diff，确认本轮执行项已全部完成或改为不采纳/阻塞，且无计划外修改。

## 侦察方向

1. 删减与收缩：死代码、无用字段、无用函数、无用分支、重复实现、过期兼容层、未使用配置、临时桥接、过度抽象、模型碎片化、DTO 爆炸。
2. 边界与依赖：绕过架构边界、依赖方向反转、跨层泄漏、模块过度耦合、公共 API 过宽、文件职责混杂或过度拆分、领域对象被基础设施污染。
3. 职责与流程：函数职责重复、参数过多或重复、无意义转发、过深调用链、绕远控制流、可合并的局部流程、同一业务规则散落多处。
4. 状态与错误面：空值泛滥、状态组合过多、隐式约定、重复校验、异常路径不一致、资源生命周期不清、异步状态与事务边界不清。
5. 理解与验证：命名不一致、领域概念漂移、测试重复或只锁实现细节、文档过期、注释解释错误抽象、示例或配置仍指向旧设计。

## 重构优先级

1. 先删除，再合并，再移动，最后才抽象。
2. 优先消除不再被当前行为需要的兼容层、迁移桥、旧入口、双路径、过期配置和防御性冗余。
3. 优先让一个事实只有一个来源，让一个领域概念只有一个名字，让一个行为只有一条主路径。
4. 只有当新增抽象能明确减少状态、分支、重复知识、公共 API 或跨模块认知负担时，才允许新增抽象。
5. 不以“以后可能需要”为理由保留框架、钩子、配置、适配层或空实现。

## 核心原则

1. 功能不变是硬约束；所有删改必须有引用、测试、行为或运行路径证据。
2. 删除优先于新增，收敛优先于扩展；避免用新抽象抵消复杂度。
3. 更少代码不等于更短代码；优先减少状态、分支、概念、层级和重复知识。
4. 更硬优先于更灵活；让非法状态不可表示，让边界、类型、接口和错误路径更明确。
5. 人类可读优先于技巧；控制流线性、命名统一、局部上下文足够理解。
6. 当前事实优先于未来想象；不为未出现的扩展保留历史遗留、框架、钩子或配置。
7. 架构收益优先于表面整洁；只接受能降低复杂度、错误面或认知负担的改动。
8. 合理架构必须表现为：模块边界稳定、依赖方向单向、公共 API 收窄、领域语言一致、状态归属清楚、错误路径可验证。
9. 每轮结果必须可验证：代码更少、更硬、更清晰，且行为不变、回滚边界清楚。
