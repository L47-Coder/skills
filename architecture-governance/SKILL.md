# 持续架构治理

## 目标

把代码库当作一个长期演化的系统，而不是一堆待整理文件。持续清除 AI 堆叠和局部补丁造成的偶然复杂度，让模块边界、事实来源、状态归属、依赖方向和错误语义逐步收敛。

治理没有固定轮数，也没有自然的“全部完成”。每轮只处理一个可以验证、回滚和交接的根因批次；完成检查点后立即开始下一轮。

启动或恢复前完整阅读 [操作协议](references/operating-protocol.md)。

## 原则

- **保持行为**：默认不改变用户可见行为、公共 API、数据格式、协议、序列化和时序。疑似 bug 先记账，除非契约证据或用户授权允许修复。
- **先看证据**：发现必须落到文件、符号、调用或数据路径、依赖、测试、运行结果或历史 diff，不能凭个人风格重构。
- **先看全局**：先理解模块职责、依赖方向、事实来源、状态所有权和上下游影响，再动局部代码。
- **真正降熵**：优先删除、合并、归位和收窄。不要用新包装层、配置、抽象或兼容路径掩盖旧问题。
- **保护用户工作**：启动时记录已有修改；只拥有本轮新增的精确改动，不能覆盖或顺带提交用户内容。
- **不制造工作**：没有高价值改动时记录检查证据，换范围或换观察角度；不要为了持续运行而制造 churn。

## 每轮怎么做

1. **观察**：核对分支、状态和工作树，从覆盖最少或风险最高的范围开始，只读检查架构、事实、状态、冗余和契约。
2. **归因**：交叉验证证据，把同一根因的症状合并，区分真正问题与个人偏好。
3. **选择**：按架构收益、证据、行为风险、改动范围和可回滚性排序，只选择一个内聚批次。
4. **建立基线**：写清必须保持的行为、当前证据、验证命令和回滚边界。证据不足时先补测试或缩小范围。
5. **实施**：按“删除 → 合并 → 移动 → 收窄 → 最后才抽象”解决根因，不顺手扩展计划。
6. **验证**：先跑定向检查，再跑受影响范围的广泛验证；复审完整 diff，确认没有新状态、新分支、新包装层或双路径。
7. **检查点**：更新发现、覆盖、架构地图和下一动作；有代码变更时形成一个可回滚提交，然后进入下一轮。

同一时间只能有一个在制批次。发现可以积累，但不能同时铺开多个半迁移重构。

## 持续关注什么

- 模块边界、依赖方向、公共 API 和基础设施泄漏；
- 重复事实、重复模型、双写、双入口和领域语言漂移；
- 状态组合、控制流、错误、事务、异步、缓存和资源生命周期；
- 不可达代码、空壳抽象、过度防御、兼容残留和 AI 复制痕迹；
- 测试、文档、配置、生成代码和真实运行契约是否一致。

安全、性能、可靠性、隐私、schema、部署和可观测性按项目事实纳入相关视角，不为凑清单机械扩张范围。

## 续跑与交接

在目标仓库维护：

- `.architecture-governance/state.md`：长期运行状态、架构地图、覆盖、发现、阻塞和下一动作；
- `.architecture-governance/plan.md`：当前轮唯一的实施批次。

每次阶段切换、长任务前、上下文可能压缩前和被迫停止前都更新状态。恢复时先用 run ID、分支、轮次、Git 检查点和工作树对账，不重新扫描已验证范围，也不重复提交。

测试通过、提交成功、工作树干净、待办暂时为空或某个范围没有发现，都只是检查点。记录结果后换范围、运行路径、数据流、依赖图或历史热点继续观察。

只有用户明确暂停/停止，或整个治理运行在尝试缩小范围、替代工具和独立切面后仍被同一真实条件阻塞，才交还控制。不要把阶段性收敛称为治理完成。
