# 持续架构治理

## 定位

把目标代码库当作一个长期演化的系统，而不是一组待清理文件。持续识别并消除 AI 一次性生成、快速堆叠和局部修补造成的偶然复杂度，让事实来源、领域语言、状态归属、依赖方向和错误路径逐步收敛。

这不是格式化、表面整洁、一次性审查或限定轮数的重构任务。显式启动治理后，治理运行没有自然的“完成”状态。

启动或恢复治理运行前，必须完整读取 [references/operating-protocol.md](references/operating-protocol.md)，并按其中的状态文件、侦察、排序和门禁协议执行。

## 不可妥协的约束

1. **外部行为不变**：默认保持用户可见行为、公共 API、持久化数据、协议、序列化和时序语义不变。疑似 bug 先进入账本；只有既有契约能够证明预期行为，或用户另行授权时才修复。
2. **证据先于判断**：每项发现都要落到文件、符号、调用或数据路径、测试、运行结果、依赖关系或历史 diff；禁止仅凭风格偏好重构。
3. **全局先于局部**：先确认模块职责、依赖方向、事实来源、状态所有权和上下游影响，再修改局部。局部更漂亮但系统边界更差的改动必须拒绝。
4. **降熵而非搬熵**：优先减少状态、分支、概念、层级、重复知识和公共表面积；禁止用新包装层、新配置或新抽象掩盖旧复杂度。
5. **无限运行、有限轮次**：治理生命周期没有自然终点；每轮却必须是能到达 `CHECKPOINT` 的有限原子事务。每轮只实施一个内聚批次，拥有明确基线、验证和回滚边界。发现可以很多，在制改动只能有一个。
6. **不制造虚假工作**：无限治理要求持续侦察，不要求为了保持忙碌而改代码。没有高价值改动时，记录证据并更换范围或切面；禁止无收益 churn。
7. **保护用户工作**：记录启动时已有的未提交文件和 hunks；这些文件默认不可拥有。只能暂存本轮相对基线新增的精确 hunks，并逐项复核 staged diff。不能证明不冲突时，转向独立范围；整个仓库都无法安全推进时才阻塞。

## 无限存活契约

1. 将一次显式启动视为持续目标，不设置轮数、时间或 token 完成条件。宿主提供持久目标或自动续轮机制时使用它，但除非用户明确给出预算，否则不自行设置预算。
2. 测试通过、提交成功、工作树干净、待办暂时为空、单一范围饱和或完成任意轮数，都只是检查点，不是终点。
3. 每轮检查点后立即进入下一轮 `SCOUT`。本轮无改动时不得创建空提交；更新覆盖账本并从新的范围、入口、依赖切面、数据流或运行路径继续。
4. 进度更新不是收尾。对用户简洁报告当前轮次、证据、风险和下一步，同时继续执行。
5. 只有以下情况可以交还控制：
   - 用户明确要求停止、暂停、改变范围或只回答问题；
   - 同一真实阻塞达到宿主规定的阻塞阈值，且尝试更小切口、替代工具、独立范围和其他侦察入口后，整个治理运行仍无法推进；
   - 系统、权限、工具或上下文边界强制中断。
6. 被迫交还控制前，必须更新持久状态，明确标记治理为 `active`、`paused`、`blocked` 或 `stopped`，并写出可直接执行的下一动作；禁止把阶段性收敛宣称为治理完成。

## 启动与恢复

1. 读取仓库内全部适用规则，确认构建、测试、静态检查、运行入口、部署边界和生成代码边界。
2. 检查 Git 状态、worktree、detached HEAD、当前分支、最近历史和用户修改；同时探测并读取 `.architecture-governance/state.md` 与 `.architecture-governance/plan.md`，此时禁止切换分支或生成新运行 ID。
3. 状态存在时，以其中的稳定 `run-id` 为准，用轮次检查点、Git trailers、当前 `HEAD` 和工作树对账，从记录的准确阶段恢复；旧版状态缺少 ID 时做兼容迁移，禁止把它误判为新运行或无故从头扫描。
4. 状态不存在时才生成短 `run-id`。需要新分支时默认使用 `codex/architecture-governance-<YYYYMMDD>-<run-id>`，不得强占其他 worktree 已检出的分支；已有用户指定的专用任务分支时沿用。
5. 确定运行模式：`audit-only`、`implement-no-commit` 或 `implement-and-commit`。服从用户和仓库约束；用户明确启动持续治理且未限制写入时，默认在专用分支使用 `implement-and-commit`。
6. 状态文件不存在时按操作协议创建：`state.md` 保存长期控制面、架构地图、覆盖矩阵和发现账本；`plan.md` 只保存当前轮次的一个执行批次。
7. 建立或刷新系统模型：模块与依赖、领域事实来源、状态与资源所有权、关键调用/数据路径、外部契约、验证命令、高风险区和近期高变更区。

## 架构师视角

持续维护以下全局问题的答案；无法回答时，优先侦察而不是立即改动：

- 系统的稳定边界在哪里，依赖是否单向，哪些公共表面正在泄漏内部结构？
- 每项业务事实由谁拥有，是否存在重复表示、双写、双路径或多个名字？
- 状态、事务、异步任务、缓存和资源生命周期由谁创建、变更、提交与释放？
- 错误在哪一层产生、转换、恢复和暴露，失败语义是否一致且可验证？
- 测试保护的是外部契约还是当前实现形状，文档和配置是否仍描述真实系统？
- 一处局部复杂度是根因，还是更上游边界、模型或所有权错误的症状？

## 轮次状态机

每轮严格按以下阶段推进，并把当前阶段写入 `state.md`：

`SCOUT → SYNTHESIZE → SELECT → BASELINE → IMPLEMENT → VERIFY → CHECKPOINT → SCOUT`

`audit-only` 模式使用 `SCOUT → SYNTHESIZE → SELECT → CHECKPOINT → SCOUT`，只更新发现与覆盖状态，不创建 active 实施批次；`implement-no-commit` 走完整状态机但保留经验证的工作区检查点，不提交。

1. **SCOUT**：先重新核对分支、检查点和工作树所有权，再根据覆盖矩阵选择最高风险或最久未检查的范围，用五个治理切面完成只读侦察。切面必须全部覆盖，但子代理数量随可用并发调整，不固定为五个。
2. **SYNTHESIZE**：主代理交叉验证侦察结论，合并同根发现，区分根因与症状，并以稳定 ID 写入发现账本。子代理结论不是事实，未经主代理复核不得实施。
3. **SELECT**：按架构收益、复发面、证据强度、行为风险、改动表面积和可回滚性排序，只选择一个内聚批次进入 `plan.md`。其余有效发现保留为 `ready`，不得丢弃或伪装成“不采纳”。
4. **BASELINE**：记录要保持的契约、当前行为证据、目标验证命令和回滚点。缺少关键行为证据时，先补特征测试或缩小范围。
5. **IMPLEMENT**：遵循“删除 → 合并 → 移动 → 收窄 → 最后才抽象”。修复根因，避免顺手处理计划外问题。
6. **VERIFY**：先运行定向验证，再运行受影响范围的广泛验证；复审完整 diff，检查计划外修改、新状态、新分支、新抽象、新兼容层和新的双路径。
7. **CHECKPOINT**：更新发现状态、覆盖矩阵、架构地图、验证结果和下一动作。存在代码变更时按仓库规则提交一个可回滚提交；只提交本轮拥有的文件。随后立即进入下一轮。

验证暴露的新问题先写入账本；只有它阻塞当前验证且属于同一根因时，才能纳入当前批次。

## 多代理侦察

1. 有可用子代理时优先并行侦察；根据并发上限分批覆盖五个切面，不因名额不足跳过切面，也不创建超过环境额度的代理。
2. 侦察代理使用最小上下文启动，不继承完整对话；提示词必须包含目标仓库、轮次范围、唯一切面、只读约束、证据要求和固定输出结构。宿主支持权限隔离时强制使用只读 sandbox；否则在启动前后比较 Git 状态和文件指纹。
3. 侦察代理禁止修改文件、扩展任务或给出无证据的“大重写”建议。无发现时要列明实际检查范围。
4. 主代理亲自负责系统建模、跨切面归因、优先级、计划、实现、验证和 diff 审查；这些决策不得外包。
5. 侦察等待使用不超过 60 秒的短轮询；连续两次无新进展即视为本次失败。某个切面失败时只重试一次，随后由主代理本地补做；仍不可完成则阻塞该 `范围 × 切面` 并切换范围，不让单一侦察失败卡死整个运行。其他切面结果可以先记账，但在当前范围五个切面齐全前不得进入 `SELECT`。
6. 等待所有本批侦察者进入终态后再整合；进入下一轮前复用或释放已完成代理。宿主没有关闭接口时，不让失效线程继续累积，也不超过当前并发额度。

## 决策与实现门禁

只有同时满足以下条件的发现才能进入当前批次：

- 有可复查的仓库证据，且根因不只是命名、格式或个人偏好；
- 能说明对全局边界、事实来源、状态空间、错误面或认知负担的净收益；
- 能描述保持不变的契约、受影响消费者和最小充分验证；
- 改动范围内没有无法隔离的用户修改；
- 回滚边界清楚，不依赖“以后再清理”的半迁移状态。

涉及公共 API、数据迁移、序列化、权限、安全、并发、事务、时间语义或用户可见行为时，缺少契约证据不得直接实施。需要不可逆操作、外部系统协调、跨仓库消费者变更或扩大授权时，只阻塞该发现并请求相应权限，继续治理其他独立范围。

## 防复发闭环

同类熵源重复出现时，不要永久重复人工清理。确认共同根因后，优先用最小的架构适应度测试、依赖/边界规则、类型约束、lint、代码模板、生成器约束或项目级 AI 编码规则阻止再次引入，并验证防线不会锁死实现细节或制造更大的维护面。治理收益同时看存量熵是否下降，以及新增同类熵的速率是否下降。

## 降熵优先级

1. 删除不可达代码、过期入口、迁移桥、空扩展点、未用配置和防御性冗余。
2. 合并重复事实、重复模型、重复校验、重复流程和语义相同的多条路径。
3. 把职责、状态和规则移回唯一所有者，恢复单向依赖和明确生命周期。
4. 收窄公共 API、DTO、可空性、状态组合、错误类型和跨模块知识。
5. 只有新增抽象能可证明地减少总概念数、分支数、状态数或依赖面时才新增；不得为假想未来保留框架。

每轮的成功标准不是“改了多少”，而是系统在行为不变的前提下变得更少、更硬、更清晰，并且下一位执行者能从状态文件无歧义续跑。
