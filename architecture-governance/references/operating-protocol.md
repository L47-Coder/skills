## 1. 持久状态

在目标仓库维护两个文件。除非仓库规则明确禁止，否则跟随治理提交一并版本化。

### `.architecture-governance/state.md`

这是跨轮次、跨上下文恢复的长期控制面。保持简洁，只保留当前事实、未关闭发现和最近检查点；已完成细节由 Git 提交保存。

```markdown
# Architecture Governance State

## Run
- Protocol: 2
- Status: active | paused | blocked | stopped
- Run ID: <stable short id>
- Mode: audit-only | implement-no-commit | implement-and-commit
- Branch: <branch>
- Checkpoint: bootstrap@<sha> | Rxxxx
- Round: R0001
- Phase: SCOUT | SYNTHESIZE | SELECT | BASELINE | IMPLEMENT | VERIFY | CHECKPOINT
- Scope: <本轮范围>
- Worktree ownership: <基线 dirty manifest/hunk fingerprints 与本轮拥有 hunks>
- Next action: <一条可直接执行的动作>

## Repository model
- Modules/boundaries: <模块、职责、依赖方向>
- Sources of truth: <核心事实及所有者>
- State/lifecycles: <状态、事务、异步与资源所有权>
- External contracts: <API、数据、协议、序列化、时序>
- Validation commands: <命令与适用范围>
- Generated/vendor boundaries: <不可直接治理的边界>

## Coverage ledger
| Scope | Lens | Last round | Evidence/outcome | Revisit trigger |
|---|---|---:|---|---|

## Findings backlog
| ID | Lens | Scope | Status | Priority | Summary | Evidence |
|---|---|---|---|---:|---|---|

## Active blockers
| Finding | Missing fact/authority/tool | Attempts | Independent work remaining |
|---|---|---|---|

## Recent checkpoints
| Round | Result | Validation | Next scope |
|---|---|---|---|
```

规则：

- `Checkpoint`、`Phase`、工作树所有权和 `Next action` 必须足以在上下文丢失后恢复。
- `Run ID` 在首次创建后保持稳定；恢复时先读取它，再定位分支和 trailers。只有确认不存在治理状态时才能生成新 ID。
- 不在同一次提交包含的状态文件中记录该提交自身的 SHA；使用轮次 ID 和 Git trailer 定位检查点，避免自引用。
- 架构地图只记已证实的当前事实，不写愿景文档。
- 覆盖账本的单元是 `范围 × 切面`，避免反复扫描同一位置却误以为在持续治理。
- 最近检查点保留少量滚动摘要；旧记录由 Git 历史承载，避免状态文件无限膨胀。
- 每次阶段切换、长工具调用前、可能压缩上下文前和被迫交还控制前更新状态。
- 只有 `IMPLEMENT` 可以修改业务文件；其他阶段只读业务文件，必要时仅更新治理状态。
- 发现旧版状态缺字段时，在当前轮检查点做向前兼容迁移并记录理由；不得重写既有治理提交或重放已验证轮次。

## 2. 发现生命周期与优先级

为发现分配稳定 ID，例如 `AG-BND-0042`。状态只允许：

- `observed`：有信号，证据或根因尚不足；
- `ready`：证据、收益、契约和验证足够，可被后续轮次选择；
- `active`：当前轮次唯一在制根因簇；
- `validated`：已实施、验证并形成检查点；
- `rejected`：经证据证明不是问题、收益为负或违背约束，必须写理由；
- `blocked`：缺少明确事实、权限、工具或用户决策，必须写解锁条件。

不得因为本轮容量不足把有效发现标为 `rejected` 或 `blocked`；保持 `ready`。治理是无限的，待办不需要塞进同一轮。

### 行为证据等级

- **E0 假设**：只有直觉或表面形状，不得实施；
- **E1 静态事实**：引用、可达性、类型或依赖证据，只适合风险很低且边界封闭的删减；
- **E2 特征证据**：现有契约测试或补充的当前行为测试；
- **E3 差分证据**：改动前后对同一输入、回放或快照的结果一致；
- **E4 契约证据**：属性、协议、schema、跨服务契约或真实运行证据。

证据要求随风险上升。反射、插件、外部消费者、运维开关、灾备和灰度路径不能仅靠仓库内“无引用”判死；公共 API、数据格式、事务、并发与资源生命周期优先要求 E3/E4。没有可靠 oracle 时，当前批次只能补观测或测试。

对每项发现用 0–3 评估：

- **架构杠杆**：能否同时减少多个模块、路径或概念的复杂度；
- **复发/故障面**：是否持续诱发复制、补丁、状态组合或错误；
- **证据强度**：静态猜测、调用证据、运行/测试证据、交叉证据；
- **行为风险**：对外部契约、数据、并发和时序的影响；
- **改动表面积**：涉及模块、消费者、迁移和验证范围；
- **可回滚性**：能否由一个内聚提交安全撤回。

排序参考为：

`架构杠杆 + 复发/故障面 + 证据强度 - 行为风险 - 改动表面积`

分数不是自动决策。优先选择高杠杆、强证据、可回滚的根因簇；不要让几个容易改的症状压过一个清晰的系统性根因。

## 3. 五个治理切面

### L1 架构拓扑、边界与依赖

检查模块职责、依赖方向、跨层调用、公共 API、数据访问边界、基础设施污染、循环依赖、部署边界与所有权空洞。寻找局部代码为何能绕过系统规则。

### L2 事实来源、领域模型与重复知识

检查同一业务事实的多份表示、DTO/模型爆炸、命名漂移、重复校验、复制算法、双写、双入口、配置与代码表达同一规则、注释维护旧语义。目标是在同一 bounded context 或外部契约内，一个事实一个所有者、一个概念一个名字、一种行为一条主路径。

### L3 状态、控制流、错误与生命周期

检查布尔旗标组合、空值泛滥、隐式状态机、绕远控制流、异常吞噬、回退链、错误转换不一致、异步/事务边界、缓存一致性、资源创建与释放。特别关注 AI 为“保险”叠加的防御分支是否隐藏真实契约。

### L4 删减、收缩与 AI 残留

检查不可达代码、无用参数/字段/配置、空壳抽象、无意义转发、过期兼容、迁移桥、半完成框架、重复帮助函数、整文件复制、过度通用化和只被测试 mock 需要的生产接口。先证明可删，再考虑改写。

### L5 契约、验证、可理解性与运行一致性

检查测试是否保护行为而非实现、文档/示例/配置是否对应当前系统、构建与运行入口是否分叉、错误和观测是否可验证、生成代码边界是否明确。命名与格式问题只有在它们造成领域歧义、边界误用或验证缺口时才升级为治理发现。

五个核心切面不是全局架构的上限。根据系统事实把安全、隐私、schema/数据一致性、性能、可靠性、可观测性、部署拓扑、供应链和 SLO 作为扩展关注点，归入最相关切面或单列扩展切面；不得因此跳过核心五切面，也不要求额外固定数量的子代理。

## 4. 侦察提示词与输出契约

每个侦察任务使用最小上下文。不要把其他侦察者的结论或期望答案泄漏给它。

```text
目标仓库：<绝对路径>
治理轮次与范围：<Rxxxx + 明确范围>
唯一治理切面：<L1-L5，附本切面定义>

只做只读侦察，禁止修改、格式化、生成或删除文件。遵守仓库规则。
从项目事实出发，优先寻找能降低系统复杂度、错误面或认知负担的根因。
每项结论必须给出文件与符号、调用/数据/依赖路径、测试或运行证据；证据不足时明确写为假设。

固定输出：
1. 实际检查范围
2. 结论摘要
3. 发现列表：证据、根因、全局影响、保持不变的契约、最小改动方向、风险、验证建议
4. 不建议改动及理由
5. 若无发现：明确写“无高价值发现”并列出已检查证据
```

主代理整合时必须：

1. 亲自打开关键文件并复核证据；
2. 追踪至少一条上游和一条下游路径，确认不是局部错觉；
3. 合并同根症状，拒绝相互矛盾或超出范围的建议；
4. 把所有有效发现写入账本，不因本轮未选择而丢弃。

## 5. 轮次计划与执行门禁

### SELECT 门禁

- 五个切面的本轮范围结论齐全；
- 选中项是一个可独立验证和回滚的根因簇；
- 已列出消费者、外部契约、数据/状态影响和计划外边界；
- 行为证据足够，或当前批次只补充观测/特征测试；
- 与用户未提交修改没有无法隔离的重叠；
- 计划不会留下双路径、半迁移或“下一轮再删旧实现”。

### IMPLEMENT 门禁

- `plan.md` 已全量重写，只有一个在制批次；
- 定向验证在改动前可运行，或失败原因已记录；
- 删除项有引用/可达性/运行证据；
- 新抽象有净减少概念、状态、分支或依赖面的证明。

### VERIFY 门禁

依次检查：

1. 与发现直接相关的定向测试、静态检查或运行路径；
2. 受影响模块及消费者的广泛验证；
3. 完整 diff 是否超出计划；
4. 是否迎合测试而破坏真实契约；
5. 是否新增包装层、状态、分支、回退、兼容、公共表面或重复事实；
6. 代码量下降是否伴随可读性和边界提升，而不是压缩成技巧代码；
7. 状态文件、计划和实际 HEAD 是否一致。

任一关键门禁失败，回到 `BASELINE`、`IMPLEMENT` 或回滚该批次，不带着已知不确定性提交。

禁止用 `reset --hard`、`clean`、自动 `stash`、`amend`、`rebase`、强制切换/删除分支或覆盖文件来通过门禁。验证命令不得未经授权触发生产数据、部署、消息发送、付费资源或其他外部副作用。

## 6. 覆盖轮换与饱和处理

每轮从覆盖账本选择高风险或最久未检查的 `范围 × 切面`。范围不只按目录划分，还要轮换以下入口：

- 模块/包与依赖图；
- 一条用户可见运行路径；
- 一条核心数据或状态生命周期；
- 公共 API、协议、schema 与序列化边界；
- 错误、重试、回退、事务与异步路径；
- Git 高变更/高返工热点与近期 AI 大块生成 diff；
- 测试到生产实现的映射和未被行为测试保护的核心路径；
- 构建、配置、代码生成、部署与运行入口；
- 跨模块重复知识和相似实现簇。

某范围无高价值发现时：

1. 记录实际证据和扫描深度；
2. 写明重新检查触发条件，例如相关文件变化、依赖升级或上游重构；
3. 选择新的范围或改变图切法，立即进入下一轮；
4. 禁止扩大低价值命名/格式问题，也禁止为了产生提交而制造抽象。

当所有已知范围暂时饱和时，重新从运行路径、数据流、依赖图或 Git 历史构建系统地图，发现新的切分方式。饱和意味着换观察坐标，不意味着完成。

## 7. Git、检查点与恢复

- 新建分支时使用 `codex/architecture-governance-<YYYYMMDD>-<run-id>`，除非用户或仓库规定其他名称。先检查 `git worktree list` 和 detached 状态，禁止强行切换到其他 worktree 正在使用的分支。
- 每轮侦察前和提交前都重新核对分支、工作树与用户修改；上游或其他代理改变基线时先对账，禁止沿用失效计划。
- 每个有代码改动的轮次形成一个内聚提交，建议消息为 `governance(Rxxxx): <降熵结果>`，并添加 `Governance-Run: <run-id>` 与 `Governance-Round: Rxxxx` trailers，供状态文件无自引用地定位检查点。
- 启动时保存基线 dirty manifest 和 hunk 指纹；基线已修改文件默认排除。确需同文件协作时，只按精确 patch/hunk 暂存，并证明 staged diff 全部来自本轮增量；禁止路径级 `git add` 混入用户 hunks。
- 不创建空提交。侦察轮只更新状态；若仓库不希望提交治理状态文件，保留工作区检查点并遵守其规则。
- 不自动推送、合并、发布或创建 PR，除非用户明确授权相应外部动作。
- 恢复时先用 run ID、分支、轮次 trailer、阶段和工作树对账，再按下表幂等恢复。

| Phase | 恢复动作 |
|---|---|
| `SCOUT` | 保留有证据的已完成切面，核对侦察基线后补齐缺失切面；基线变化则重做受影响切面。 |
| `SYNTHESIZE` | 从原始侦察证据重建账本变更，重新复核关键文件，不重复创建 finding ID。 |
| `SELECT` | 核对 `ready` 状态和依赖，重新生成唯一 active 批次，不沿用已失效排序。 |
| `BASELINE` | HEAD、依赖或环境变化时重跑行为基线；否则复用有命令、退出码和证据的结果。 |
| `IMPLEMENT` | 先审查现有 diff 与 owned hunks，只继续未完成步骤；禁止盲目重放修改。 |
| `VERIFY` | 按记录的命令重新运行必要验证，区分基线已有失败与本轮回归。 |
| `CHECKPOINT` | 先按 `Governance-Run`/`Governance-Round` trailer 查找提交；存在则只前移状态，不重复提交。 |

## 8. 阻塞处理

阻塞一项发现不等于阻塞整个治理运行。

1. 记录缺失的事实、权限、工具或决策，首次出现位置、已尝试方法和解锁条件。
2. 缩小改动范围、补充只读证据、替换工具或转向不依赖该条件的范围。
3. 涉及行为选择时只阻塞该发现，继续处理其他 `ready` 项或开展新侦察。
4. 只有同一阻塞达到宿主要求的重复阈值，且所有独立范围都无法推进时，才把运行标记为 `blocked`。
5. 被迫交还控制时，状态必须说明治理未完成、最后可靠检查点、未验证改动、阻塞证据和下一条可执行动作。
