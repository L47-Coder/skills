# 持续架构治理操作协议

这份协议负责让治理能够长期运行、随时恢复，并且每一轮都安全、有限、可验证。

## 1. 状态文件

除非仓库明确禁止，在目标仓库维护并版本化两个文件。

### `.architecture-governance/state.md`

保存长期事实，不写流水账：

```markdown
# Architecture Governance State

## Run
- Protocol: 2
- Status: active | paused | blocked | stopped
- Run ID: <stable id>
- Mode: audit-only | implement-no-commit | implement-and-commit
- Branch: <branch>
- Round / phase / scope: <当前检查点>
- Worktree ownership: <基线用户修改与本轮拥有的 hunks>
- Next action: <一条可直接执行的动作>

## Architecture
- Modules and dependencies: <职责与依赖方向>
- Sources of truth: <核心事实与所有者>
- State and lifecycles: <状态、事务、异步与资源归属>
- External contracts: <API、数据、协议、序列化和时序>
- Validation: <命令与适用范围>
- Generated/vendor boundaries: <不可直接治理的内容>

## Coverage
| Scope | Lens | Last round | Evidence/outcome | Revisit trigger |
|---|---|---:|---|---|

## Findings
| ID | Lens | Scope | Status | Priority | Summary | Evidence |
|---|---|---|---|---:|---|---|

## Blockers
| Finding | Missing fact/authority/tool | Attempts | Independent work remaining |
|---|---|---|---|
```

保持 `Run ID` 稳定。状态必须足以在上下文丢失后恢复；旧检查点细节交给 Git 历史，不让文件无限增长。每次阶段切换、长工具调用前和被迫停止前更新。

### `.architecture-governance/plan.md`

只保存当前轮的一个根因批次，每轮重新写：

```markdown
# Governance Round R0001
- Scope / why now: <范围和优先原因>
- Findings: <稳定 ID，一个根因簇>
- Current problem: <现状与证据>
- Entropy to remove: <要减少的状态、分支、概念或依赖>
- Invariants: <必须保持的行为>
- Implementation: <最小步骤>
- Validation: <定向、广泛和 diff 检查>
- Rollback: <回滚边界>
- Out of scope: <本轮明确不做>
- Next action: <检查点后的动作>
```

## 2. 启动和恢复

1. 读取仓库规则、构建、测试、静态检查、运行入口、部署边界和生成代码边界。
2. 检查 Git 状态、worktree、detached HEAD、分支、近期历史和用户修改，再读取已有治理状态；此时不要切分支或创建新 run ID。
3. 有状态时用其中的 run ID、轮次、Git trailers、HEAD 和工作树恢复。缺字段的旧状态只做向前兼容，不重放已验证轮次。
4. 没有状态时才创建 run ID。需要专用分支时使用 `codex/architecture-governance-<YYYYMMDD>-<run-id>`，但不强占其他 worktree 的分支。
5. 确认模式：`audit-only`、`implement-no-commit` 或 `implement-and-commit`。用户启动持续治理且未限制写入时，默认在专用分支实施并提交。
6. 建立初始架构地图和覆盖表：模块、依赖、事实来源、状态与资源所有权、外部契约、验证命令、高风险区和近期高变更区。

## 3. 一轮治理

### 观察

从风险最高或最久未检查的范围开始，覆盖五个视角：

1. **边界与依赖**：模块职责、依赖方向、跨层调用、公共 API、基础设施污染和循环依赖。
2. **事实与领域**：重复模型、重复校验、双写、双入口、命名漂移和同一规则的多份表达。
3. **状态与流程**：布尔组合、空值、隐式状态机、错误转换、事务、异步、缓存和资源生命周期。
4. **删减与 AI 残留**：不可达代码、无用字段、空壳抽象、转发层、兼容桥、复制实现和过度通用化。
5. **契约与验证**：测试是否保护行为，文档/配置/生成代码是否对应真实系统，运行入口和错误语义是否一致。

每项发现都要给出文件或符号、调用/数据/依赖路径以及测试、运行或历史证据。证据不足就标为假设，不实施。

### 归因和选择

- 主代理亲自复核关键文件，至少追踪一条上游和一条下游路径。
- 合并同根症状，把有效发现写入账本；本轮没选中的保持 `ready`，不能丢弃或伪装成拒绝。
- 优先选择高架构收益、强证据、行为风险低、表面积小、可独立回滚的根因簇。
- 只允许一个 `active` 批次。缺少行为证据时，本轮只补观测或特征测试。

发现状态使用：`observed`、`ready`、`active`、`validated`、`rejected`、`blocked`。`rejected` 必须说明为什么不是问题或收益为负；`blocked` 必须写清缺失条件。

### 建立基线和实施

- 写清外部行为、公共 API、数据、序列化、并发、事务、权限和时序中必须保持的部分。
- 先运行能描述当前行为的测试或路径。没有可靠 oracle 时，缩小范围或只补测试。
- 实施顺序是：删除、合并、移动、收窄，最后才新增抽象。
- 新抽象必须证明减少了总概念、状态、分支或依赖面；不要为假想未来搭框架。
- 不顺手处理计划外问题。验证中发现的新问题先记账，只有同一根因且阻塞当前验证时才纳入。

### 验证和检查点

依次检查：

1. 与发现直接相关的测试、静态检查或运行路径；
2. 受影响模块及消费者的广泛验证；
3. 完整 diff 是否超出计划或混入用户修改；
4. 是否新增了包装层、状态、分支、回退、兼容、公共表面或双路径；
5. 代码减少是否真的改善边界和理解成本，而不是压成技巧代码；
6. 状态、计划、HEAD 和验证结果是否一致。

关键检查失败就回到基线、实施或回滚，不能带着已知不确定性提交。检查点完成后更新状态并进入下一轮。

## 4. Git 和用户修改

- 启动时记录 dirty 文件和 hunk 指纹。已有修改默认不归治理任务所有。
- 只暂存本轮新增的精确 hunks，并逐项复核 staged diff；不要用路径级暂存混入用户内容。
- 每个有代码变更的轮次形成一个内聚提交，建议使用 `governance(Rxxxx): <降熵结果>`，并添加 `Governance-Run` 和 `Governance-Round` trailers。
- 不创建空提交，不自动 stash，不使用 `reset --hard`、`clean`、`amend`、`rebase`、强制切换/删除分支或覆盖文件。
- 不自动推送、合并、发布或创建 PR，除非用户明确授权。
- 验证命令不得未经授权触发生产数据、部署、消息发送、付费资源或其他外部副作用。

## 5. 可选的并行侦察

有可用子代理时，可以把五个视角拆成只读侦察任务并行执行；没有时由主代理本地完成，不能跳过视角。

- 每个侦察任务只拿到目标仓库、轮次范围、一个视角、只读约束和证据格式，不泄漏其他侦察者结论。
- 侦察者只能报告检查范围、证据、根因、影响、风险和验证建议，不能修改文件或发起大重写。
- 主代理负责复核、跨视角归因、排序、实施、验证和 diff 审查，不能把架构决策外包。
- 侦察失败时重试一次，再由主代理补做；单个视角不能卡死整个长期运行，但当前范围五个视角不齐时不能实施。

## 6. 没有发现时

没有高价值改动不代表治理完成：

1. 记录实际检查范围、证据深度和重新检查触发条件；
2. 换一个模块、运行路径、数据流、依赖切面、公共契约、错误路径或 Git 热点；
3. 所有已知范围暂时饱和时，从运行路径、数据流、依赖图或历史重新切分系统地图；
4. 不把命名、格式和低价值抽象放大成工作，也不创建空提交。

同类问题反复出现时，治理防复发机制：最小边界测试、依赖规则、类型约束、lint、模板、生成器约束或项目级 AI 编码规则。防线应阻止熵源，而不是锁死实现细节。

## 7. 阻塞和交接

阻塞一项发现不等于阻塞整个运行。记录缺失事实、权限、工具或决策，尝试缩小范围、补证据、替换工具或转向独立范围。

只有同一阻塞达到宿主要求的重复阈值，而且所有独立范围都无法推进时，才把运行标为 `blocked`。用户要求暂停/停止，或系统强制中断时，也必须先更新状态，写清最后可靠检查点、未验证改动和下一条可执行动作。

测试通过、提交成功、工作树干净、待办暂时为空和单一范围饱和都只是检查点。不要把阶段性收敛宣称为治理完成。
